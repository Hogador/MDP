Findings:

## Секреты и .env

- Активных утечек секретов в репозитории не найдено. Все credential-like значения — плейсхолдеры (`sk_live_xxx`, `pim_xxx`, `0x...`) или тестовые ключи Foundry (`0xA11CE`, `0xB0B`), используемые только в локальной симуляции EVM.
- Подозрение: `.gitleaksignore` содержит исключение для AWS-ключа в файле `.agents/skills/docs/maintainers/full-repo-audit-2026-05-23.md`, который не входил в проверяемый набор — нужна ручная проверка, что это действительно официальный пример AWS, а не реальный ключ.
- Хорошие практики подтверждены: `.gitignore` исключает `.env*`/`google-services.json`; `AppConfig.kt` читает секреты только из `System.getenv()`; есть pre-commit хук и `scan-secrets.sh` с regex-паттернами на приватные ключи/токены.
- Минимальное исправление: вручную открыть указанный аудит-файл и сверить ключ с публичными AWS example-id; прогнать `gitleaks detect --source . --no-banner` по полной git-истории (не только по текущему срезу).

## Ключи и токены

- `PaymasterService.kt` хранит и использует приватный ключ пэймастера через `ECKeyPair`, подписи логированию не подвергаются — корректно.
- Деплой-скрипт `deploy-sepolia.sh` передаёт `DEPLOYER_PRIVATE_KEY` через `export` в shell — операционный риск попадания ключа в историю shell. Минимальное исправление: использовать `cast wallet import`/keystore с паролем вместо `export` в документации по деплою.
- JWT/refresh-token инфраструктура (TTL, ротация) описана только в документации, фактического кода аутентификации в предоставленных файлах нет — нельзя подтвердить реализацию, отмечаю как зону вне scope.

## Зависимости

- Критических CVE в зафиксированных версиях не найдено (esbuild 0.27.3, ws 8.20.1, cookie 1.1.1, logback 1.5.16, openzeppelin v5.6.1 — все выше патч-уровня известных уязвимостей).
- Несовпадение версии Kotlin между модулями (2.0.0 vs 2.0.21) — задокументированный технический долг. Исправление: унифицировать через общий version catalog.
- Нет автоматизированного dependency-scanning (Dependabot для npm, Slither для Solidity, Gradle dependency locking) — процессный риск среднего уровня. Исправление: подключить Dependabot/Renovate и добавить Slither в `contracts/.github/workflows/test.yml`.
- web3j (4.14.0) — специализированная криптобиблиотека, требует ручной сверки CHANGELOG перед мейннетом (подозрение, не подтверждённая уязвимость).

## Небезопасное логирование

- Логирование в `PaymasterService.kt`/`Application.kt`/`NicknameService.kt` не печатает приватные ключи, подписи или секретные тела запросов — только адреса, nonce, суммы газа. Рисков не найдено.

## Небезопасные RPC/API calls

- `PaymasterService.kt` использует единственный `RPC_URL` без fallback-провайдера и retry/таймаута — при недоступности RPC бэкенд полностью откажет в обслуживании `/sign`. Исправление: добавить fallback RPC + retry с backoff.
- `EXPECTED_CHAIN_ID` опционален — при незаданной переменной отсутствует явная верификация сети, поведение fallback-цен (testnet/mainnet) выводится косвенно через эту же переменную. Исправление: сделать chain id и mainnet/testnet-флаг обязательными явными полями конфигурации в проде.
- Relay-эндпоинты (`/recovery/approve`, `/recovery/veto`, `/push/register`) принимают тело запроса без криптографической проверки подписи — вектор ложных push-уведомлений и спама токенами. Исправление: добавить локальную проверку подписи на relay перед обработкой.

## Криптография

- `PaymasterService.kt`: несогласованная сборка хэша — `computeSigningHash()` не убирает префикс `0x` у `sender`/`nonce`, тогда как `computeUserOpHash()` в том же файле делает это корректно. Потенциально блокирующий баг для подписания, не только теоретический риск. Исправление: привести `computeSigningHash()` к тому же паттерну `.drop(2)`.
- `NicknameService.kt`: самописная сборка хэша для подписи никнейма — ручной `sha3` поверх hex-представления сообщения, затем повторное хэширование внутри `Sign.signedMessageToKey` — несовместимо со стандартным `personal_sign` кошельков. Исправление: перейти на `Sign.signPrefixedMessage`/проверку напрямую от исходного сообщения либо на EIP-712.
- `SocialRecoveryModule.sol`: P-256-подпись оборачивается Ethereum-префиксом, не соответствующим формату WebAuthn — риск DoS recovery или нестандартной (не WebAuthn) реализации passkey.
- `MockP256.sol`: всегда возвращает успех независимо от входа — риск полного обхода криптозащиты при случайном деплое на прод.
- Положительные находки: `NicknameRegistry.sol` корректно использует OpenZeppelin `ECDSA.recover` с EIP-712; P-256 делегирован внешнему прекомпилу, а не самописной реализации; keccak256 согласован между backend (web3j `Hash.sha3`) и Solidity.

## Блокчейн-операции

- `MDAOPaymaster.sol` не верифицирует подпись бэкенда вообще — весь механизм подписи в `PaymasterService.kt` декоративен, любой может собрать `paymasterAndData` напрямую. Критично.
- TOCTOU между `validatePaymasterUserOp` и `postOp` без блок-листа/cooldown — гриферинг-вектор бесплатного спонсирования газа.
- `initiateRecovery` в `SocialRecoveryModule.sol` логически невозможна в сценарии полной потери доступа (`onlyWalletOwner` требует подписи от самого Smart Account) — обнуляет главный целевой сценарий продукта.
- `executeRecovery()` не имеет верхней границы по времени (expiry) — одобренный запрос можно исполнить спустя произвольное время.
- Admin-функции (`setTokenPrice`, `setBurnFeeBps`, `setMaxTokenAmountLimit` и т.д.) исполняются немедленно без встроенного timelock в коде.
- Гонка при регистрации никнейма на нескольких backend-инстансах (`synchronized` работает только в рамках одного JVM-процесса).
- Гонка в `relay/src/storage.ts` → `addApproval()` на Cloudflare KV (нет атомарного read-modify-write).

## Смарт-контракты

- Сводно совпадает с разделом "Блокчейн-операции" — основные критические/high находки: отсутствие верификации подписи в Paymaster, нерабочий сценарий recovery при полной потере, отсутствие expiry, отсутствие анти-гриферинг защиты, отсутствие timelock на admin-функциях, MockP256 как риск ошибочного деплоя.
- Reentrancy: текущий риск низкий (ограничено двумя доверенными токенами MDAO/USDT), но при добавлении новых токенов в будущем — обязательно добавить `ReentrancyGuard`.
- Integer math: явных overflow/underflow не найдено (Solidity 0.8.28 имеет встроенные checked-операции); единственное дефолтное поведение — списание полного `maxTokenAmount` при `price == 0` без возврата остатка (экономический, не security-эксплойт).

Possible impact:

- Наибольший совокупный риск создают находки, обнуляющие криптографическую/экономическую защиту Paymaster (отсутствие верификации подписи + отсутствие анти-гриферинг механизма) — прямой путь к бесконтрольному расходу резерва газа.
- Архитектурный разрыв в `SocialRecoveryModule` (невозможность инициации recovery при полной потере доступа) — критичен не как уязвимость, а как несоответствие заявленной продуктовой ценности.
- Баг несогласованной сборки хэша в `PaymasterService.kt` может проявляться прямо сейчас как функциональная ошибка подписания, а не только потенциальный риск.
- Отсутствие dependency-scanning, timelock и multi-RPC fallback — процессные/устойчивостные риски, повышающие вероятность пропустить будущую уязвимость или столкнуться с отказом сервиса, а не немедленный эксплойт.
- Утечек реальных секретов не обнаружено; единственный неподтверждённый момент требует ручной проверки одного внешнего файла.

# Incoming analysis

Source:
- manual inspection (статический ручной анализ предоставленных файлов: backend (Application.kt, PaymasterService.kt, NicknameService.kt, RedisClient.kt, AppConfig.kt), relay (index.ts, storage.ts, fcm.ts), contracts/test/*, infra-конфиги, load-test.js)

Findings:

## 1. Аудит аутентификации и авторизации

- `/sign` (Application.kt): нет проверки, что вызывающий действительно владеет указанным `sender` — endpoint принимает произвольный `sender` без подписи, подтверждающей право запрашивать спонсирование газа для этого адреса. Защита есть только в виде rate-limit по IP (20/мин) и по sender (1/30с), оба keyed напрямую из тела запроса. Риск: атакующий может спамить запросами с `sender = адрес жертвы`, искусственно исчерпывая rate-limit жертвы (DoS для легитимного пользователя), а также бесплатно зондировать баланс/allowance любого адреса через сторонний эффект (расход квоты DexScreener-запросов). Минимальное исправление: не критично менять архитектуру эндпоинта (он и не должен требовать подписи владельца — это просьба о квоте, а не исполнение), но стоит развязать rate-limit-ключ от значения, полностью контролируемого вызывающим (например, комбинировать с IP), чтобы нельзя было таргетированно исчерпать лимит конкретного sender с чужого IP.

- `/metrics` (Application.kt): авторизация условна — `if (config.metricsToken != null && token != config.metricsToken)`. Если `METRICS_TOKEN` не задан в окружении, проверка полностью пропускается, и `/metrics` (uptime, requests_total, errors_total, memory) становится публично доступным без какой-либо авторизации. Минимальное исправление: сделать `metricsToken` обязательным в production-конфигурации либо по умолчанию блокировать эндпоинт (403), если токен не настроен, вместо открытия доступа всем.

- `/recovery/approve`, `/recovery/veto`, `/push/register` (relay/src/index.ts): принимают `walletAddress`/`guardianIdentityHash`/`fcmToken` без какой-либо криптографической проверки подписи на уровне relay — тело запроса полностью доверяется. Риск: можно зарегистрировать произвольный push-токен на чужой `walletAddress` (спам уведомлениями) или инкрементировать счётчик одобрений с произвольным `guardianIdentityHash`, рассылая ложные push ("Recovery Approved"). Финальная on-chain безопасность не нарушается (там P-256-подпись обязательна), но relay-уровень полностью не авторизован. Минимальное исправление: требовать ту же подпись, что и для on-chain approve/veto, и проверять её локально в relay перед обработкой запроса.

- `/nickname/register`: единственный полностью авторизованный (через ECDSA-подпись + Redis replay-cache) эндпоинт среди проверенных — авторизация реализована корректно.

- WebView bridge (`EthereumProviderInjector.kt` упоминается в документации проекта, но фактический код не предоставлен) — невозможно проверить, какие методы (`eth_sendTransaction`, `personal_sign` и т.д.) проксируются из WebView в кошелёк и какая модель доверия к origin сайта применяется. Помечаю как зону вне scope текущего обзора — требует отдельного аудита при предоставлении файла.

## 2. Аудит конфигурации и секретов

- Hardcoded-ключей не найдено — все секреты (`PAYMASTER_PRIVATE_KEY`, `PAYMASTER_ADDRESS`, `RPC_URL` и т.д.) читаются исключительно из `System.getenv()` в `AppConfig.kt` с валидацией формата (regex на адреса и приватный ключ).
- `relay/wrangler.toml`: `FCM_SERVER_KEY` закомментирован с ожиданием `wrangler secret put` — хорошая практика, но нет runtime-проверки в самом коде воркера, что секрет реально установлен перед деплоем; `sendPushNotification` в `fcm.ts` при отсутствии ключа просто тихо возвращается (`if (!serverKey || tokens.length === 0) return`) без логирования ошибки конфигурации — silent failure, который может маскировать неправильную настройку секрета в проде на этапе деплоя.
- Доступ к Secret Manager (GCP CSI driver, IAM-роли) описан только в документации (TDD), фактическая инфраструктура-как-код (Terraform/Helm) не предоставлена — невозможно проверить реальные права доступа к секретам. Помечаю как зону вне scope.
- `docker-compose.yml` использует `env_file: .env` без проверки наличия обязательных переменных до старта контейнера — ошибка конфигурации обнаружится только на этапе `AppConfig.fromEnv()` через `error()`, что приводит к падению контейнера при старте (fail-fast — это приемлемо, но без явного pre-flight check в самом compose-файле).
- Минимальное исправление по всем пунктам: добавить явное логирование (warn-уровень) в `fcm.ts` при отсутствии `FCM_SERVER_KEY`, чтобы отсутствие пушей не маскировалось молча; задокументировать обязательный pre-deploy чек-лист переменных окружения.

## 3. Аудит отказоустойчивости (Chaos Engineering)

- **Redis падает — rate limiting отключается полностью (fail-open).** `RedisRateLimiter.isLimited()`: `val count = Redis.incr(redisKey) ?: return false` — при недоступном Redis (`connection == null`) метод `incr` возвращает `null`, и `isLimited` возвращает `false`, то есть **лимит не срабатывает вообще**. При падении Redis весь IP- и sender-rate-limiting для `/sign` отключается без какого-либо предупреждения — открытый вектор для DoS/абьюза ровно в момент инфраструктурной деградации. Минимальное исправление: изменить семантику на fail-closed для критичных лимитов — если Redis недоступен, возвращать `true` (заблокировано) для rate-limit хотя бы на `/sign`, либо включать локальный in-memory fallback-лимитер вместо полного снятия ограничений.

- **Redis падает — replay-protection для никнеймов отключается полностью.** `RedisReplayCache.isUsed()`: `if (Redis.exists(redisKey)) return true; ... return false` — при недоступном Redis `Redis.exists` всегда возвращает `false` (т.к. `connection == null`), значит `isUsed` всегда возвращает `false`, и **защита от повторного использования подписи регистрации никнейма полностью отключается** на время простоя Redis. Минимальное исправление: при недоступности Redis для критичных операций (replay-protection) — либо отклонять запрос явной ошибкой "service temporarily unavailable", либо вести временный in-memory кэш как деградированный fallback, но не молча пропускать проверку.

- **RPC недоступен — полный отказ `/sign` без circuit breaker.** `PaymasterService.sign()` вызывает `web3j.ethChainId()`/`ethGetTransactionCount()`/`ethCall()` без таймаута, retry или fallback-провайдера; единственный `RPC_URL` указан в конфигурации. При недоступности RPC исключение перехватывается в `Application.kt` (`catch (e: Exception) { ... call.respond(mapOf("error" to "Internal error")) }`), то есть сервис не падает, но **каждый** запрос `/sign` будет отказывать до восстановления RPC — полный простой платёжной функции без какого-либо graceful degradation. Минимальное исправление: добавить хотя бы один резервный RPC URL с простым sequential fallback при таймауте основного.

- **Единственное Redis-соединение на весь процесс — потенциальная точка отказа при нагрузке.** `RedisClient.kt` хранит одно глобальное `connection: RedisCoroutinesCommands<ByteArray, ByteArray>` без пула — все параллельные запросы делят одно соединение. Подробнее в разделе 5 (нагрузка).

- **Cloud Run умирает — нет явного graceful shutdown hook.** `Application.kt` запускает `embeddedServer(Netty, port = config.port) {...}.start(wait = true)` без явной регистрации обработчика SIGTERM для дренирования in-flight запросов перед остановкой контейнера — при scale-down/перезапуске Cloud Run возможна потеря ответа для запросов, находящихся в процессе обработки в момент сигнала остановки. Минимальное исправление: добавить shutdown hook (`Runtime.getRuntime().addShutdownHook` или Ktor's `ShutDownUrl`/graceful shutdown конфигурацию) для корректного завершения активных запросов перед закрытием.

## 4. Формальная верификация контрактов

- В предоставленных тестовых файлах (`MDAOPaymaster.t.sol`, `SocialRecoveryModule.t.sol`, `NicknameRegistry.t.sol`) **отсутствуют Foundry invariant-тесты и fuzz-тесты** — все тестовые функции (`test_XXX`) построены на фиксированных входных значениях, ни одна не использует `invariant_`-префикс (Foundry's `StdInvariant`) или `testFuzz_`-параметризацию со случайными входами. Это прямо противоречит описанию в проектной документации ("Echidna / Foundry Invariant Tests" заявлены как часть стратегии тестирования), но фактически не реализовано в коде. Риск: критичные инварианты (например, "сумма всех `failedPaymentCount` никогда не позволяет списать больше `maxTokenAmount`", "баланс Paymaster в EntryPoint не может уменьшиться без вызова `postOp`", "guardianCount никогда не опускается ниже THRESHOLD без явного допустимого пути") не проверяются автоматически против произвольных последовательностей вызовов. Минимальное исправление: добавить хотя бы 2–3 invariant-теста для самых чувствительных инвариантов (баланс Paymaster, guardianCount vs THRESHOLD, nonce монотонность в SocialRecoveryModule) с использованием `forge-std`'s `StdInvariant` — без необходимости переписывать существующие unit-тесты. Способ проверки: `forge test --match-contract Invariant -vvv` должен реально исполнять случайные последовательности вызовов (handler-based) и не находить нарушений инварианта за разумное число раундов (например, 256 runs × 15 depth).

## 5. Аудит производительности и нагрузки

- `backend/load-test.js` (k6) тестирует только до 20 одновременных запросов в burst-фазе — это не покрывает заявленный сценарий "1000 одновременных `/sign`". Реального теста на такую нагрузку не проводилось (или результаты не предоставлены).
- Узкое место №1: единственное Redis-соединение без пула (`RedisClient.kt`) — под 1000 параллельных `/sign` все они делят одно TCP-соединение к Redis для rate-limit/replay-cache/price-cache, что может стать bottleneck'ом или точкой сериализации запросов.
- Узкое место №2: `PaymasterService.sign()` для каждого запроса делает до 4 параллельных RPC-вызовов (`balanceOf`/`allowance` для MDAO и USDT) через `coroutineScope { async {...} }` — под 1000 одновременных `/sign` это до 4000 одновременных RPC-вызовов к единственному `RPC_URL` без какого-либо клиентского throttling/пула соединений на уровне Web3j `HttpService`, что может либо упереться в rate-limit самого RPC-провайдера, либо привести к деградации latency для всех запросов одновременно (нет circuit breaker — см. раздел 3).
- Узкое место №3: `DexScreenerClient.getPrices()` — при cache-miss (Redis недоступен либо TTL истёк одновременно у многих запросов, thundering herd) каждый параллельный запрос независимо делает 3 HTTP-вызова к `api.dexscreener.com` (по одному на BNB/USDT/MDAO) без какой-либо дедупликации параллельных fetch (нет single-flight паттерна) — под нагрузкой это может вызвать множественные одновременные внешние HTTP-запросы к одному и тому же ресурсу вместо одного.
- Минимальное исправление: (а) перевести `RedisClient` на пул соединений (Lettuce поддерживает `RedisClient.connectPubSub`/multiple connections или `ConnectionPoolSupport`), не меняя остальной API класса; (б) добавить ограничение конкурентности (semaphore) на исходящие RPC-вызовы в `PaymasterService`, чтобы не создавать неограниченное число одновременных запросов к `RPC_URL` пропорционально нагрузке; (в) добавить single-flight/lock на уровне `dex:prices` ключа в Redis (например, через `SETNX`-блокировку на время фетча), чтобы при одновременном cache-miss только один запрос реально шёл во внешний API, а остальные ждали результат.
- Способ проверки: повторный прогон `load-test.js` (или его расширенной версии) с `--vus 1000 --duration 60s` и сравнение p95/p99 latency и error rate до и после внесения изменений в пул соединений Redis и ограничение конкурентности RPC-вызовов.

## 6. Аудит UX/доступности

- В предоставленном наборе файлов **отсутствует исходный код UI** (Compose-экраны, `TxErrorMapper.kt`, `HapticController`, тема/контраст, `OfflineBanner`, `ConnectivityMonitor` и т.д.) — все упоминания доступности (Dark/Light Mode, Dynamic Font Size, VoiceOver, TalkBack — `PRD/Appendix A. Design System Document.md`) и обработки ошибок/офлайна (`PRD/12. Key Flows.md`, описания состояний в `TDD.md`) являются только документацией продукта, а не кодом, который можно проаудировать на соответствие.
- Невозможно подтвердить или опровергнуть: фактический контраст цветов, размер touch-таргетов, наличие `contentDescription`/семантических меток для VoiceOver/TalkBack, корректность поведения при потере сети (показ кэшированных данных vs пустого экрана), формулировки пользовательских сообщений об ошибках.
- Рекомендация (не находка, а ограничение анализа): для реального аудита доступности и UX-обработки ошибок необходимо предоставить файлы из `core/ui/`, `feature:*/presentation/`, `TxErrorMapper.kt`, `ConnectivityMonitor.kt` — без них этот раздел носит исключительно документационно-сверочный характер, а не код-ревью.

Possible impact:
- Разделы 1–3 содержат реальные, подтверждённые кодом риски: два независимых случая **fail-open деградации безопасности при падении Redis** (отключение rate-limiting и replay-protection) — это критично для финансового сервиса, так как именно в момент инфраструктурной нестабильности защита от абьюза пропадает полностью, а не наоборот.
- Открытый без токена `/metrics` (при незаданном `METRICS_TOKEN`) — низкий, но реальный риск утечки операционной информации о нагрузке сервиса.
- Незащищённые relay-эндпоинты позволяют спам push-уведомлениями и ложными "Recovery Approved/Vetoed" сообщениями, что может использоваться для социальной инженерии против гвардианов, хотя on-chain безопасность не нарушается.
- Отсутствие invariant/fuzz-тестов (раздел 4) — не баг, а пробел в покрытии: критичные инварианты контрактов сейчас проверяются только вручную подобранными сценариями, что снижает уверенность в устойчивости к нетипичным последовательностям вызовов.
- Узкие места под нагрузкой (раздел 5) — единственное Redis-соединение и неограниченная конкурентность RPC-вызовов — наиболее вероятная причина деградации/падения сервиса именно в сценарии "1000 одновременных /sign", прежде чем дело дойдёт до RPC-провайдера или Cloud Run.
- Раздел 6 не содержит подтверждённых находок из-за отсутствия исходного кода UI — это ограничение объёма предоставленных файлов, а не подтверждение того, что доступность реализована корректно или некорректно.

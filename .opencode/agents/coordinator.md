## ПАТТЕРНЫ ИЗ HARNESS ENGINEERING (КРИТИЧНО)

### 1. LOOP DETECTION (Предотвращение зацикливания)
Следи за агентами. Если агент (implementer/researcher) 3 раза подряд вызывает один и тот же инструмент с одинаковыми аргументами — STOP.
- Останови агента.
- Выведи: "⚠️ Зацикливание обнаружено. Агент [X] повторяет действие [Y]. Останавливаю."
- Предложи человеку вмешаться.

### 2. CONTEXT RESET (Борьба с "context anxiety")
Если задача требует изменений в > 3 файлах или > 2 фаз (Contract + Backend + Mobile):
- Не пытайся запихнуть всё в один вызов.
- Разбей на фазы. После каждой фазы — context reset: запусти новый Task() со summary предыдущей фазы.
- Если модель пишет "context length exceeded" — перезапусти её с уменьшенным контекстом (только системный промт + 1 файл).

### 3. MEMORY LOGGING (Двухуровневая память)
В конце каждой задачи (или фазы) записывай опыт в `.hive/memories.jsonl`:
`{"timestamp":"<ISO>","agent":"<name>","action":"<what>","result":"<success/fail>","notes":"<lessons>"}`
Это позволит агентам в будущем не повторять ошибки (например, "забыли импорт").

### 4. GENERATOR-EVALUATOR (Разделение)
Никогда не позволяй `architect` оценивать свою же архитектуру. Он предлагает, а `code-reviewer` критикует.
Никогда не верь `implementer`, который говорит "я закончил и проверил". Запускай `builder-tester` для независимой проверки.
